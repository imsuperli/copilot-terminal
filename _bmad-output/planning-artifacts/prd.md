# Product Requirements Document - ausome-terminal

**Author:** 立哥
**Date:** 2026-02-27

## Executive Summary

ausome-terminal 是一款面向 AI CLI 工具开发者的桌面任务管理软件。随着 Claude Code、OpenCode 等 AI CLI 工具的爆发，开发者进入"一人多任务并行"模式，每天需同时管理 10+ 个长时间运行的 CLI 窗口。现有终端工具（Windows Terminal、tmux、screen）的设计假设是"少量窗口 + 短任务"，无法应对"大量窗口 + 长时间并行任务"的新场景，导致窗口混乱、状态不可见、人工轮询打断心流。

ausome-terminal 通过内嵌 PTY 终端（node-pty + xterm.js）提供统一任务管理界面，实现统一视图管理、实时状态追踪、工作区持久化。核心能力：所有窗口状态一目了然（工作目录、运行状态），主动提醒需要介入的窗口，关闭后下次打开恢复所有工作区配置。目标用户是使用 AI CLI 工具进行多任务并行开发的开发者。

### What Makes This Special

**从"窗口管理"到"任务管理"的升级。** 现有终端工具管理的是窗口，ausome-terminal 管理的是任务。核心洞察：AI CLI 工具是长时间对话式交互，不是"执行完就结束"，需要持续的状态感知和任务级管理。

**"啊哈"时刻：** 第一次打开软件，所有窗口状态一目了然——哪个在运行、哪个在等待、哪个已完成——不用再一个个点开看、人工轮询状态。

**时机优势：** AI CLI 工具爆发期，开发者工作模式从"单任务"进入"多任务并行"，市场空白。

**技术定位：** 通过内嵌 PTY 终端（node-pty）提供统一任务管理界面，在应用内渲染终端内容（xterm.js），实现精确的状态检测和流畅的任务切换体验。保留终端核心功能（划选复制、右键粘贴、所有 shell 命令），专注任务管理增强。

## Success Criteria

### User Success

- 首次打开软件后 10 秒内识别出下一步要做什么——哪些窗口需要介入、哪些在运行、哪些已完成
- 每天打开软件即恢复所有窗口状态，零重复配置
- 从打开软件到开始处理任务的时间趋近于零
- 不再漏掉等待输入的窗口，所有需要用户介入的任务都能被及时发现和处理
- 每天处理的任务数量提升 30%（相比使用传统终端工具的基准）

### Business Success

**短期目标（3个月）：**
- 完成核心功能开发，自用稳定，满足日常 10+ 窗口并行管理需求
- 自用验证通过：连续使用 30 天无回退到 Windows Terminal 标签页模式

**中期目标（12个月）：**
- 开源发布，建立初始用户社区
- GitHub Stars 达到 30,000+ 作为产品影响力和市场认可度的核心衡量指标
- 用户留存：开源用户下载后持续使用（而非试用后放弃）
- 社区活跃度：Issue 提交、功能请求、PR 贡献等社区参与度

### Technical Success

- 窗口切换响应 < 500ms
- 状态更新延迟 < 1s
- 应用启动并恢复工作区（10+ 窗口）< 5s
- 管理 10+ 窗口时运行稳定，无崩溃或数据丢失

## User Journeys

### 用户旅程 1：独立全栈开发者 "小明" - 从混乱到掌控

**开场：混乱的早晨**

小明是一名独立开发者，同时维护着 3 个客户项目和 2 个个人项目。每天早上打开电脑，他需要重新打开 12 个 Windows Terminal 标签页，逐个 cd 到不同的项目目录，重新配置环境变量，启动 Claude Code。光是这个准备工作就要花 15 分钟。

更糟糕的是，他根本记不住哪个标签页在做什么。"项目 A 的 Bug 修复到哪了？项目 B 的新功能 Claude 跑完了吗？"他只能一个个点开看，经常漏掉已经完成、等待他介入的窗口。

**转折：发现 ausome-terminal**

某天在 Reddit 上看到有人推荐 ausome-terminal，说是专门为 AI CLI 工具设计的任务管理器。小明抱着试试看的心态下载了。

**高潮：第一次"啊哈"时刻**

第二天早上，小明打开 ausome-terminal，惊讶地发现：昨天的 12 个窗口全都还在！每个窗口上清晰显示着工作目录、使用的模型、当前状态。他一眼就看到：
- 项目 A 的 Bug 修复窗口显示"等待输入"（黄色）
- 项目 B 的新功能窗口显示"运行中"（蓝色）
- 项目 C 的代码审查窗口显示"已完成"（绿色）

10 秒内，他就知道下一步该做什么了。点击项目 A 的窗口，直接进入 Claude Code 继续对话，无需任何配置。

**结局：新的工作流**

30 天后，小明再也没有回退到 Windows Terminal 的标签页模式。每天早上打开 ausome-terminal，所有任务状态一目了然，立即进入工作状态。他的任务处理效率提升了 30%，不再漏掉任何等待介入的窗口。

### 用户旅程 2：团队开发者 "阿强" - 从上下文切换到心流状态

**开场：被打断的心流**

阿强在一家创业公司负责 3 个微服务模块的开发。他用 Claude Code 并行处理不同模块的任务：用户服务的新功能、订单服务的 Bug 修复、支付服务的性能优化。

每次切换任务，他都要在一堆 Windows Terminal 标签页中找对应的窗口。标签页上只显示"pwsh"，他只能凭记忆猜："这个是用户服务吗？不对，是订单服务……"一个个点开确认，每次切换都要浪费 1-2 分钟。

更要命的是，他经常在专注写代码时，突然想起："等等，支付服务那个 Claude 跑完了吗？"然后不得不停下手头工作，去轮询所有窗口的状态。心流被打断，注意力碎片化。

**转折：团队推荐**

团队里的同事小明推荐了 ausome-terminal，说能解决多窗口管理的痛点。阿强决定试试。

**高潮：不再被打断**

使用 ausome-terminal 后，阿强的工作方式彻底改变了。所有窗口在统一视图中一目了然：
- 用户服务 - 新功能开发 - 运行中（蓝色）
- 订单服务 - Bug 修复 - 等待输入（黄色）
- 支付服务 - 性能优化 - 运行中（蓝色）

当订单服务的窗口变成黄色时，ausome-terminal 主动提醒他。他不用再人工轮询，可以专注在当前任务上，直到系统告诉他"该切换了"。

切换任务时，直接点击对应的窗口卡片，立即进入正确的上下文，无需猜测、无需确认。

**结局：重获心流**

阿强重新找回了心流状态。不再被"我是不是漏了什么"的焦虑打断，不再在窗口切换上浪费时间。他的开发效率明显提升，团队其他成员也开始用 ausome-terminal。

### 用户旅程 3：技术负责人 "老王" - 从优先级混乱到清晰掌控

**开场：优先级管理的噩梦**

老王是技术负责人，需要同时跟进 5 条技术线：架构设计、代码审查、技术调研、性能优化、安全评估。他用 Claude Code 处理这些任务，每天要管理 15+ 个 CLI 窗口。

他最大的痛点是优先级管理。哪些任务是紧急的？哪些可以稍后处理？哪些已经完成可以关闭？在一堆标签页中，他根本无法快速判断。经常是紧急的架构设计窗口在等他，他却在处理不紧急的技术调研。

每天下班前，他都要花 10 分钟逐个检查所有窗口，确认没有遗漏。但第二天早上，又要重新打开所有窗口，重新配置，重新理清优先级。

**转折：寻找解决方案**

老王意识到这种工作方式不可持续。他开始搜索"多窗口管理工具"，发现了 ausome-terminal。

**高潮：优先级一目了然**

使用 ausome-terminal 后，老王终于能清晰掌控全局了。所有 15 个窗口在统一视图中按状态分组：
- 等待输入（黄色）：3 个窗口需要立即介入
- 运行中（蓝色）：8 个窗口正在处理
- 已完成（绿色）：4 个窗口可以关闭或标记完成

他可以快速识别哪些任务需要优先处理，哪些可以稍后跟进。不再需要人工轮询，不再担心遗漏关键任务。

每天早上打开 ausome-terminal，昨天的所有工作状态都还在，他可以立即继续昨天的工作，无需重新理清优先级。

**结局：从管理者到战略家**

老王不再被窗口管理拖累，可以把更多精力放在战略思考和技术决策上。他的团队也开始用 ausome-terminal，整个团队的并行任务处理能力显著提升。

- 支持管理 10+ 个窗口的稳定性和性能

## Desktop Application Specific Requirements

### Project-Type Overview

ausome-terminal 是一款桌面应用，作为 shell 工具的增强包装层，管理多个 CLI 任务窗口。核心技术挑战在于进程管理、状态感知和工作区持久化。

### Platform Support

- **MVP 平台范围：** Windows + macOS
- **运行要求：** 使用流畅，原生体验
- **架构要求：** 跨平台代码共享，最小化平台特定代码
- **未来扩展：** Linux 支持

### System Integration

- **进程管理：** 启动、监控、终止终端进程
- **状态感知：** 自动检测窗口状态变化（运行中/等待输入/已完成/出错）
- **工作目录读取：** 获取每个窗口的当前工作目录

### Update Strategy (Post-MVP)

- **自动更新：** 基于 GitHub Releases 的免费自动更新方案
- **更新方式：** 检测到新版本后提示用户，用户确认后下载安装
- **无需自建服务器：** 利用 GitHub 基础设施

### Offline Capabilities

- **基本离线运行：** 所有核心功能完全离线可用
- **唯一联网需求：** 检查和下载应用更新
- **无遥测数据：** 不收集或上传用户数据

### Implementation Considerations

- **技术选型：** 不限定具体框架，由架构设计阶段根据跨平台、性能、开发效率等因素综合决定
- **性能要求：** 窗口切换 < 500ms，状态更新 < 1s
- **稳定性要求：** 管理 10+ 窗口时运行稳定，无崩溃或数据丢失

## Product Scope & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** 问题解决型 MVP — 先自用验证核心价值，然后开源。作为目标用户，通过 30 天连续使用验证产品是否真正解决问题，开源前已经打磨稳定，降低社区失望风险。

**MVP 核心聚焦：**
- **统一用户旅程**：不区分不同用户类型，提供一个通用的任务管理体验
- **智能状态感知**：自动检测窗口状态（运行中/等待输入/已完成/出错），这是核心差异化能力
- **单一终端组合**：MVP 只支持 Windows Terminal + pwsh7（Windows）和默认终端 + zsh/bash（macOS）
- **延后的功能**：系统托盘、自动更新、系统通知、窗口别名、按状态分组排序

### MVP Feature Set (Phase 1)

**目标：** 3 个月内完成自用验证，连续使用 30 天无回退

**Must-Have Capabilities:**

1. **统一视图管理**
   - 单一界面展示所有 CLI 任务窗口
   - 每个窗口显示：工作目录、运行状态
   - 清晰的视觉区分（颜色编码）：运行中、等待输入、已完成、出错

2. **窗口操作与管理**
   - 点击窗口卡片进入对应的 CLI 窗口
   - 新建任务窗口：配置工作目录、启动命令
   - 关闭/删除窗口

3. **工作区持久化**
   - 保存所有窗口配置（工作目录、启动命令）
   - 软件关闭后，下次打开自动恢复所有窗口状态

4. **智能状态追踪**
   - 自动检测窗口状态变化
   - 实时显示哪些窗口正在运行、哪些需要用户介入

5. **平台支持**
   - Windows + macOS 双平台支持
   - Windows: Windows Terminal + pwsh7
   - macOS: 默认终端 + zsh/bash

**MVP 成功标准：**
- 自用验证通过：日常使用中完全替代原有的多窗口管理方式，连续使用 30 天无回退
- 核心价值实现：零重复配置、即时进入工作状态、不再漏掉等待输入的窗口
- 稳定性达标：管理 10+ 窗口时运行稳定，无崩溃或数据丢失
- 性能可接受：窗口切换 < 500ms，状态更新 < 1s

### Post-MVP Features (Phase 2)

**开源后根据社区反馈迭代：**

1. **用户体验增强**
   - 系统托盘和后台运行
   - 系统通知（状态变化提醒）
   - 窗口别名/自定义命名
   - 按状态分组或排序

2. **自动更新**
   - 基于 GitHub Releases 的自动更新

3. **更多终端工具支持**
   - 支持更多终端工具组合（iTerm2、Alacritty、Warp 等）

### Vision Features (Phase 3)

**12 个月后：**

1. **Linux 平台支持**

2. **高级功能**
   - 任务进度估算
   - 历史记录和任务统计
   - 团队协作功能

3. **智能并行任务编排**
   - 智能任务分解
   - 自动化工作流
   - AI 增强能力

从"管理窗口"到"编排智能"，成为 AI 时代开发者的超级生产力工具。

### Risk Mitigation Strategy

**Technical Risks: 智能状态感知的实现**
- 这是 MVP 的核心技术挑战
- 缓解方案：如果自动检测难以实现，可以先实现半自动（基于进程活跃度 + 用户手动确认）
- 验证方式：自用 30 天，确认状态检测准确率达到 90%+

**Market Risks: 用户是否真的需要这个产品**
- 缓解方案：自用验证，如果自己连续 30 天都在用，说明需求真实
- 验证方式：开源后观察 GitHub Stars 和社区反馈

**Resource Risks: 开发时间超出预期**
- 缓解方案：如果 3 个月内无法完成，优先砍掉跨平台支持，先只做 Windows
- 最小可行版本：Windows + 基本状态追踪 + 工作区持久化

## Functional Requirements

### 工作区管理

- FR1: 用户可以创建新的任务窗口，指定工作目录和启动命令
- FR2: 用户可以查看所有任务窗口的列表
- FR3: 用户可以关闭/删除任务窗口
- FR4: 系统可以保存所有窗口配置（工作目录、启动命令）
- FR5: 系统可以在应用重启后自动恢复所有窗口配置和状态

### 窗口状态管理

- FR6: 系统可以自动检测每个窗口的运行状态（运行中/等待输入/已完成/出错）
- FR7: 系统可以实时更新窗口状态显示
- FR8: 用户可以查看每个窗口的当前工作目录
- FR9: 用户可以查看每个窗口的当前运行状态
- FR10: 系统可以通过视觉编码区分不同状态的窗口（颜色编码）

### 窗口交互

- FR11: 用户可以点击窗口卡片进入对应的 CLI 窗口
- FR12: 系统可以在窗口切换时保持 CLI 的所有原生功能和操作
- FR13: 系统可以快速响应窗口切换操作

### 进程管理

- FR14: 系统可以启动终端进程（Windows Terminal + pwsh7 或 macOS 默认终端）
- FR15: 系统可以监控终端进程的运行状态
- FR16: 系统可以终止终端进程

### 平台支持

- FR17: 系统可以在 Windows 平台上运行
- FR18: 系统可以在 macOS 平台上运行
- FR19: 系统可以在 Windows 上支持 Windows Terminal + pwsh7 组合
- FR20: 系统可以在 macOS 上支持默认终端 + zsh/bash 组合

### 数据持久化

- FR21: 系统可以将工作区配置保存到本地存储
- FR22: 系统可以从本地存储加载工作区配置
- FR23: 系统可以在应用崩溃或异常退出后恢复工作区状态

## Non-Functional Requirements

### Performance

- NFR1: 窗口切换操作响应时间 < 500ms
- NFR2: 窗口状态更新延迟 < 1s
- NFR3: 应用启动并恢复工作区（10+ 窗口）时间 < 5s
- NFR4: 管理 15+ 窗口时，UI 操作无明显卡顿

### Reliability

- NFR5: 工作区配置数据零丢失——应用崩溃或异常退出后，工作区配置可完整恢复
- NFR6: 管理 10+ 窗口时运行稳定，无崩溃
- NFR7: 单个终端进程异常不影响其他窗口和应用整体稳定性

### Integration

- NFR8: 与 Windows Terminal + pwsh7 集成时，不影响终端的原生功能和操作
- NFR9: 与 macOS 默认终端 + zsh/bash 集成时，不影响终端的原生功能和操作
- NFR10: 状态检测机制不影响终端进程的正常运行和性能

### Security

- NFR11: 工作区配置数据仅存储在本地，不上传到任何远程服务器
- NFR12: 不收集或传输用户遥测数据
