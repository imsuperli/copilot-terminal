# 画布内终端生命周期问题梳理（2026-05-07）

## 1. 用户诉求原文整理

当前希望把“画布里新建的终端”和“普通主页终端”彻底区分开：

1. 在画布中创建的终端，不应该在主页卡片中显示。
2. 这些终端不是“引用外部终端”，而是“画布自有终端”，生命周期应和画布一致。
3. 现在重启软件后，即使画布从未再次打开，左侧边栏仍会一直显示画布 logo，这会造成干扰。
4. 未启动的画布不应该出现在左侧边栏。
5. 已启动的画布需要支持“停止”。
6. 停止后，画布应能再次打开。
7. 打开后的画布需要支持“销毁运行态资源”。
8. 销毁时，画布内部自有终端对应的进程资源必须同步销毁。
9. 销毁运行态资源后，后续仍应可以重新打开该画布。

## 2. 建议先统一的概念

当前这几个概念在语义上容易混在一起，建议先明确：

1. `引用终端`
   说明：把一个已经存在的普通终端放进画布块里，只是引用，不归画布托管。

2. `画布自有终端`
   说明：在画布里直接新建出来的终端，归该画布托管，不应进入主页普通终端集合。

3. `画布工作区记录`
   说明：持久化的画布数据本身，包含块布局、便签、链接、默认目录等。

4. `画布运行态`
   说明：这个画布当前是否被启动、是否出现在左侧边栏、是否有需要管理的内部资源。

5. `停止`
   说明：关闭画布运行态，但保留画布记录，后面可以再次打开。

6. `删除画布`
   说明：删除画布记录本身，通常是不可恢复的业务删除。

7. `销毁运行态资源`
   说明：释放当前画布运行中的内部终端/会话/进程，但不删除画布记录。

如果不先拆开这几个概念，后面“停止”和“销毁”很容易实现成重复动作。

## 3. 现状代码观察

### 3.1 画布里新建终端目前直接进入全局 `windows`

当前 `CanvasWorkspaceView` 在画布中创建新终端时，直接：

- 调用 `createCanvasWindowDraft(...)`
- 再调用 `addWindow(draftWindow)`

这意味着画布里新建的终端，当前在数据层就是普通全局窗口。

相关位置：

- `src/renderer/components/CanvasWorkspaceView.tsx`
- `src/renderer/utils/canvasWindowFactory.ts`

### 3.2 主页卡片天然会把这些终端显示出来

主页 `CardGrid` 的窗口数据来源是全局 `windows`，只过滤了：

- `archived`
- `ephemeral`
- SSH 绑定类特殊窗口

但没有“画布自有终端”这个概念，所以画布里创建的终端会进入主页卡片。

相关位置：

- `src/renderer/components/CardGrid.tsx`
- `src/renderer/utils/sshWindowBindings.ts`

### 3.3 左侧边栏当前按“存在画布记录”显示，不按“画布是否启动”显示

当前 `Sidebar` 中的画布列表来自：

- `canvasWorkspaces.filter((canvasWorkspace) => !canvasWorkspace.archived)`

也就是只要画布记录存在，就会出现在边栏画布列表里。
这和“未启动的画布不显示”相冲突。

相关位置：

- `src/renderer/components/Sidebar.tsx`

### 3.4 当前没有独立的“画布运行态”

现有只有：

- `canvasWorkspaces`
- `activeCanvasWorkspaceId`
- `ViewSwitcher` 当前视图

但没有：

- `canvas started / stopped`
- `canvas runtime destroyed`
- `canvas-owned windows are running / stopped`

因此现在的“打开画布”更像是视图切换，不是生命周期切换。

相关位置：

- `src/renderer/stores/windowStore.ts`
- `src/renderer/hooks/useViewSwitcher.ts`
- `src/main/services/ViewSwitcher.ts`

### 3.5 当前“live / summary”只是展示模式，不是生命周期动作

画布块上的 `displayMode: 'summary' | 'live'` 只是切换嵌入展示。
`window-live-opened` / `window-live-closed` 也只是活动事件，不会创建或销毁终端资源。

这和“启动画布”“停止画布”“销毁内部进程资源”不是一回事。

相关位置：

- `src/renderer/components/CanvasWorkspaceView.tsx`

## 4. 这件事真正缺的核心模型

要满足目标，至少还缺两类元数据。

### 4.1 窗口归属

窗口至少要能区分：

- 普通窗口
- 画布引用窗口
- 画布自有窗口

建议最少补一类字段，例如：

```ts
interface Window {
  ...
  ownerType?: 'standalone' | 'canvas-owned';
  ownerCanvasWorkspaceId?: string | null;
}
```

注意：

- `引用终端` 不应改成 `canvas-owned`
- 只有“在画布里直接新建”的终端才归画布托管

### 4.2 画布运行态

画布本身需要独立运行态，例如：

```ts
interface CanvasWorkspaceRuntimeState {
  workspaceId: string;
  startedAt: string;
  status: 'running' | 'stopped';
}
```

或者更简化：

```ts
startedCanvasWorkspaceIds: string[]
```

否则边栏无法准确判断“该显示哪些画布”。

## 5. 建议的目标行为

### 5.1 主页卡片

1. 普通终端卡片里不显示 `canvas-owned` 窗口。
2. 画布卡片仍然正常显示，因为画布记录本身需要可进入。
3. 画布里的“引用终端”仍然保留原有普通终端行为。

### 5.2 左侧边栏

1. 只显示“已启动的画布”。
2. 软件重启后，默认不因为持久化记录而自动显示画布 logo。
3. 画布只有在用户主动打开/启动后，才进入边栏。
4. 画布停止后，从边栏消失。

### 5.3 画布自有终端

1. 画布内新建的终端归当前画布托管。
2. 它们不进入主页普通终端卡片。
3. 它们的进程资源在“停止画布 / 销毁运行态”时统一清理。
4. 下次重新打开画布时，允许再次启动这些终端。

## 6. 建议状态机

### 6.1 画布工作区记录状态

- `exists`
- `deleted`

这是业务记录层。

### 6.2 画布运行态状态

- `stopped`
- `running`

这是运行时层。

### 6.3 画布自有终端状态

可沿用现有 pane/window status：

- `completed`
- `restoring`
- `running`
- `waiting`
- `error`

但前提是这些窗口被明确标记为 `canvas-owned`。

## 7. 建议流程

### 流程 A：创建画布

1. 创建 `CanvasWorkspace` 记录。
2. 默认状态为 `stopped`。
3. 主页可见画布卡片。
4. 左侧边栏不显示。

### 流程 B：在画布中创建“自有终端”

1. 在画布里点“新建终端”。
2. 创建一个 `canvas-owned` window。
3. 该 window 记录只归当前画布管理。
4. 该 window 不进入主页普通终端卡片。
5. 该 window 是否立即启动，需要单独定义。

建议：

- 创建记录可以立即完成
- 进程可延迟到“启动画布”或“首次打开该块 live/terminal”时再拉起

### 流程 C：把现有终端引用到画布

1. 用户选择已有终端加入画布。
2. 只创建 block，不创建新 window。
3. 生命周期仍归原普通终端，不归画布。
4. 停止/销毁画布时，不能误杀这类被引用终端。

### 流程 D：启动画布

1. 用户从主页卡片或其他入口打开画布。
2. 画布进入 `running`。
3. 左侧边栏出现该画布项。
4. 必要时恢复/启动该画布自有终端的运行态。

这里要做一个产品决策：

1. 进入画布就启动所有自有终端
2. 进入画布只启动画布 runtime，具体终端按需启动

建议优先第 2 种，成本更低，也更符合大画布场景。

### 流程 E：停止画布

1. 用户在边栏或画布页点击“停止”。
2. 画布运行态从 `running` 变为 `stopped`。
3. 清理该画布所有 `canvas-owned` 窗口的进程/会话资源。
4. 引用进来的普通终端不受影响。
5. 左侧边栏移除该画布项。
6. 画布记录和块布局仍保留。

### 流程 F：重新打开画布

1. 用户再次打开画布卡片。
2. 画布重新进入 `running`。
3. 自有终端按策略重新启动。
4. 上一次保留的布局、块、便签继续可用。

### 流程 G：删除画布

1. 用户删除画布记录。
2. 如果画布当前处于 `running`，先执行一次运行态销毁。
3. 删除该画布所有 `canvas-owned` window 记录。
4. 删除对应 blocks / activity / template 关联数据。
5. 被引用进来的普通终端不删除。

## 8. 目前最容易出错的逻辑点

### 8.1 “停止”和“销毁”现在语义重叠

如果“停止画布”已经会把内部进程全杀掉，那它本质上已经包含“销毁运行态资源”。
这时再加一个“销毁”按钮，就必须明确它到底是：

1. 仅销毁运行态资源
2. 删除画布记录
3. 删除画布和自有终端记录

否则用户和代码都会混淆。

建议：

- `停止` = 关闭运行态，保留记录
- `删除画布` = 删除记录，且先清运行态

不建议再单独引入一个和 `停止` 高度重叠的“销毁”概念，除非你明确把它定义成“删除画布”。

### 8.2 必须区分“引用终端”和“画布自有终端”

这是整件事最关键的边界。
否则一旦停止画布，就可能把引用进来的普通终端也杀掉。

### 8.3 需要定义“启动画布”是否等于“启动全部内部终端”

这个如果不提前定，后续边栏状态、启动速度、资源占用都会来回改。

建议：

- `启动画布` 先只定义为“画布运行态启动”
- 画布内自有终端按需启动

### 8.4 删除 block 时是否删除自有终端记录

这里也要明确：

1. 删除一个画布自有终端块，是否等于删除那个终端？
2. 还是只删 block，终端记录仍保留在画布中但失去入口？

建议：

- 对 `canvas-owned` 而言，删除块就应删除其对应 window 记录和资源
- 因为它不是引用关系，block 本身就是它的唯一业务入口

### 8.5 模板实例化也要继承这套归属语义

现在模板应用时会批量 `addWindow(windowItem)`。
如果模板里创建的是画布内部终端，这些窗口也应该被标成 `canvas-owned`，否则一样会跑到主页卡片里。

## 9. 推荐落地规则

### 必须成立的规则

1. 只有 `standalone` 窗口进入主页普通终端卡片。
2. 只有 `running` 的画布进入左侧边栏。
3. 只有 `canvas-owned` 窗口会被画布停止动作统一销毁资源。
4. `引用终端` 永远不受画布停止/删除影响。
5. 删除画布前，必须先清理该画布自有终端资源。

### 推荐但待确认的规则

1. 软件重启后，不自动恢复画布运行态。
2. 打开画布不必立即启动全部内部终端，按需启动更稳。
3. 删除 `canvas-owned` block 时，直接同步删除对应 window 记录和资源。

## 10. 结论

你的目标本身是成立的，方向没有问题。
真正的问题不是单点 UI bug，而是当前数据模型里缺少两个核心维度：

1. `这个终端到底是不是画布自有的`
2. `这个画布到底有没有启动`

只要这两个维度补齐，主页卡片显示、边栏显示、停止/重开、删除时释放进程资源，都会变成可推导的行为。

当前最大的逻辑风险点有三个：

1. `停止` 和 `销毁` 的语义还没拆开
2. `引用终端` 和 `画布自有终端` 还没拆开
3. `画布存在` 和 `画布运行中` 还没拆开

如果后续要改实现，建议先补建模，再改 UI 过滤和资源清理逻辑。
