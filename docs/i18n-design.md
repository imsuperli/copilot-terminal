# 国际化设计方案

## 目标

- 为渲染层 UI 增加中英文切换能力。
- 语言设置可持久化，重启后保持用户选择。
- 首次启动时按系统语言生成默认值：中文系统默认中文，其他默认英文。
- 不引入新的第三方国际化依赖，保持当前构建链路简单可控。

## 范围

- 本次覆盖 Electron 渲染层的核心可见界面：主界面、终端视图、设置页、快捷导航、批量创建、错误提示等。
- 本次不处理终端内应用输出，也不翻译项目数据本身（例如窗口名、项目路径、外部链接标题）。

## 方案概览

### 1. 语言模型

- 支持语言：`zh-CN`、`en-US`
- 新增共享语言定义：统一做语言归一化与默认值处理。

### 2. 设置持久化

- 在工作区设置 `settings` 中新增 `language` 字段。
- 首次创建工作区时，由主进程基于 `app.getLocale()` 写入默认语言。
- 加载历史工作区时，对缺失的 `language` 做补齐与归一化，保证老数据兼容。

### 3. 渲染层架构

- 新增轻量 `I18nProvider` 与 `useI18n()` Hook。
- 文案集中放在一份翻译表中，使用 key 访问，不把中英文散落在组件里。
- 提供参数替换能力，满足 `创建 {count} 个窗口` 这类动态文案。

### 4. 时间与状态本地化

- 相对时间继续基于 `date-fns`，但按当前语言切换 locale。
- 窗口/窗格状态文案改为通过翻译 key 输出，而不是写死中文。

### 5. 用户入口

- 在设置页新增 `通用 / General` 标签页。
- 在该页提供语言选择器，切换后立即生效并异步持久化。

## 设计取舍

### 为什么不引入 `i18next`

- 当前仅需中英文两套文案。
- 项目已有 Electron + React + Zustand + Vitest 组合，增加外部依赖收益不高。
- 自研轻量实现足以满足现阶段需求，同时便于后续替换。

### 为什么把语言存到工作区设置

- 现有设置链路已经完整：`renderer -> preload -> ipc -> WorkspaceManager`。
- 复用现有存储路径，改动小，回滚成本低。
- 与 IDE、快捷导航、StatusLine 等设置一起持久化，符合用户预期。

## 兼容性

- 历史 `workspace.json` 没有 `language` 字段时，自动补全，不破坏现有数据。
- 未命中的语言值统一归一化为支持范围内的语言。
- 渲染层在设置尚未加载完成前使用中文兜底，避免启动期空白或报错。

## 实施步骤

1. 新增共享语言定义与翻译表。
2. 扩展工作区设置并补齐迁移逻辑。
3. 增加渲染层 `I18nProvider`。
4. 改造核心组件文案与状态文案。
5. 在设置页增加语言切换入口。
6. 补充测试：默认语言、英文渲染、工作区默认值/迁移。

## 验证点

- 默认工作区能根据系统语言生成 `language`。
- 历史工作区加载后能自动补齐 `language`。
- 设置页切换语言后，主界面与弹窗立即更新。
- 英文模式下，空状态、创建窗口、快捷切换、终端工具栏等关键流程文案正确。
