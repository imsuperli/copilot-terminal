# 手动分屏长命令排查记录

## 当前状态

- 问题已修复。
- 最终根因、修复方式与影响范围见：
  - `docs/手动分屏长命令问题复盘.md`

## 问题现象

- 在终端界面手动点击“左右拆分”后，左侧 pane 会出现一大段 PowerShell 命令。
- 命令内容包含 `Set-Location -LiteralPath`、`CLAUDECODE`、`TMUX`、`AUSOME_TMUX_RPC`、`@anthropic-ai\\claude-code\\cli.js`、`--agent-id` 等字段。

## 已尝试且已回滚的修复

### 1. 给手动创建的 pane 显式关闭 tmux 兼容

- 改动范围：
  - `src/main/types/process.ts`
  - `src/shared/types/electron-api.ts`
  - `src/main/services/ProcessManager.ts`
  - `src/main/handlers/paneHandlers.ts`
  - `src/main/handlers/processHandlers.ts`
  - `src/main/handlers/windowHandlers.ts`
  - `src/main/services/WorkspaceRestorer.ts`
- 思路：
  - 为 `spawnTerminal()` 增加 `enableTmuxCompat`。
  - 手动开的 window/pane/restore 路径都传 `false`。
  - 只让 `TmuxCompatService` 自己创建的 tmux pane 传 `true`。
- 结果：
  - 用户已实测，无效果。
  - 这组改动已全部回滚。

### 2. 拦截 Claude agent init 的 `send-keys`，直接替换 placeholder shell

- 改动范围：
  - `src/main/services/TmuxCompatService.ts`
  - `src/main/services/TmuxCompatService.test.ts`
- 思路：
  - 检测 `cd ... && env ... claude-code ... --agent-id ...`。
  - 不再把长命令写入 PTY，而是直接杀掉占位 shell，改为直接启动 Claude 子进程。
- 结果：
  - 用户已实测，无效果。
  - 说明之前并没有精确命中真实故障链路，或链路判断本身就是错的。
  - 这组改动已回滚。

## 当前确认的事实

- 这段长命令的文本形态，和 `TmuxCompatService.handleSendKeys()` 在 Windows 上把 Unix 风格 `env ...` 命令转成 PowerShell 语句后的结果一致。
- 因此出现该文本时，必须发生了以下两件事中的至少一件：
  - 某条 tmux `send-keys` 请求真的被发到了兼容层。
  - 或者已经被转译后的长命令真的被写进了某个 PTY，然后又被回显到了界面。

## 当前排查策略

- 不再继续猜测式修复。
- 只增加定点日志，回答两个问题：
  1. 手动 `split-pane` 时，新 pane 是否真的被当成 tmux pane 初始化。
  2. 那条长命令最终被写入/回显到的 `windowId/paneId/pid` 到底是哪一个。

## 日志文件

- 手动分屏专项日志：
  - `%TEMP%\\copilot-terminal-manual-split-debug.log`
- tmux 兼容层日志：
- `%TEMP%\\copilot-terminal-tmux-debug.log`

## 2026-03-13 新日志确认的根因

- 手动分屏发生时，左侧旧 pane 会再次调用 `get-pty-history`。
- 这次 `get-pty-history` 返回的数据里明确包含 `\x1b[c`，也就是终端 Device Attributes 查询。
- 随后左侧旧 pane 立刻通过 `pty-write` 把 `\x1b[?1;2c` 回写进了 PTY，`metadata.source = xterm.onData`。
- 与此同时，tmux 兼容层日志里没有新的 `send-keys`。

结论：

- 真正的问题不是“手动分屏时 tmux 又发了一次长命令”。
- 真正的问题是“左侧旧 pane 的同会话 history replay 又把旧的 `\x1b[c` 喂给了 xterm，xterm 再异步生成 `\x1b[?1;2c` 回写到 shell”。
- 现有 `suppressPtyWriteRef` 只在 replay 写入当下短暂生效，挡不住真实 xterm 稍后才发出的 synthetic reply。
- PowerShell / PSReadLine 在收到这串协议回复后，把历史中的旧 Claude agent 启动命令当作预测文本重新显示出来，于是看起来像“左 pane 被写入了一长串命令”。
