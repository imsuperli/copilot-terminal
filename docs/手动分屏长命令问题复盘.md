# 手动分屏长命令问题复盘

## 问题现象

- 在终端界面手动执行左右分屏后，左侧原 pane 会突然出现一长串 PowerShell 命令。
- 命令内容看起来像 Claude teammate 初始化命令，包含 `Set-Location -LiteralPath`、`CLAUDECODE`、`TMUX`、`AUSOME_TMUX_RPC`、`@anthropic-ai\claude-code\cli.js`、`--agent-id` 等字段。

## 最终根因

- 根因不在 tmux 兼容层，也不是手动分屏时又触发了一次 tmux `send-keys`。
- 真正的问题在渲染层 `TerminalPane` 的 history replay 路径。
- 手动分屏时，左侧旧 pane 又执行了一次同会话 `get-pty-history`。
- 这份 history 里包含终端握手查询 `ESC[c` 一类 Device Attributes 查询。
- xterm 在重放这段历史时，会把这类查询当成真实终端协议请求处理，并异步生成 `ESC[?1;2c` 等 synthetic reply。
- 这些 synthetic reply 又通过 `xterm.onData -> ptyWrite` 被写回左侧旧 pane 的 shell。
- PowerShell / PSReadLine 收到这串协议回复后，会把历史中的旧 Claude 初始化命令当作预测文本重新显示，于是界面上看起来像“手动分屏把长命令写进了左侧 pane”。

## 为什么之前的修复无效

- 给手动分屏关闭 tmux 兼容无效：
  - 因为复现时并没有新的 tmux `send-keys` 发生。
- 拦截 Claude agent init 的 `send-keys` 无效：
  - 因为问题不是“当场发送了长命令”，而是“旧历史里的协议查询被 replay 后重新触发了协议回复”。

## 最终修复

- 修复位置：
  - `src/renderer/components/TerminalPane.tsx`
- 修复方式：
  - 在 history replay 时，移除 `ESC[c`、`ESC[>c`、`ESC[=c` 这类 Device Attributes 查询。
  - 保留普通可见输出的回放，不再把终端握手查询重新喂给 xterm。

## 影响范围

- 直接影响范围：
  - `TerminalPane` 的 history replay 逻辑。
  - 任何会触发同会话历史重放的场景，如果历史里带有 Device Attributes 查询，都可能出现同类问题。
- 本次已确认的触发场景：
  - 手动左右分屏时左侧旧 pane 的重放。
- 高风险相邻场景：
  - 同一个 pane 在不更换会话的前提下被重新挂载或重新请求 history 的场景。
  - 例如某些布局切换、视图切换、恢复显示、组件重建。
- 更容易出现可见异常的终端：
  - PowerShell，尤其是启用了 PSReadLine 预测/建议显示时。
- 不受这次问题直接影响的部分：
  - tmux RPC server 本身。
  - tmux `send-keys` 的正常执行链路。
  - 新开 PTY 的实时启动输出。
  - 普通终端文本历史回放。

## 修复后的行为边界

- 历史回放仍然用于恢复可见屏幕内容。
- 只有终端握手查询不会再被回放。
- 实时 PTY 输出里的协议查询仍然保留，不影响真实终端启动握手。

## 结论

- 这是一个“终端协议查询被错误地当成可重放屏幕内容”的问题，不是 tmux 命令误发问题。
- 修复后，手动分屏不会再因为 replay 旧的 `ESC[c` 查询而把 synthetic reply 写回左侧 shell，也就不会再把旧 Claude 初始化命令显示出来。
