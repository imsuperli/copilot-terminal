# 2026-04-10 性能走查与升级风险评估

## 1. 背景

目标是提升以下用户体验性能：

- 打开应用和首次进入终端的速度
- 终端长时间运行后的输入流畅度
- 窗口切换、分屏切换、组视图切换时的顺滑度
- 后台窗口、隐藏窗口、SSH 窗口长期运行时的资源占用

本次只做走查、归纳和升级评估，不改业务逻辑。

关联分支：

- `chore/perf-upgrade-assessment-20260410`

## 2. 当前版本基线

代码库当前版本：

- `electron`: `28.3.3`
- `electron-builder`: `24.13.3`
- `react`: `18.2.0`
- `vite`: `5.0.0`
- `node-pty`: `1.1.0`
- `@xterm/xterm`: vendored custom `6.0.0`

2026-04-10 当天查到的上游版本：

- `electron`: `41.2.0`
- `electron-builder`: `26.8.1`
- `react`: `19.2.5`
- `vite`: `8.0.8`
- `node-pty`: `1.1.0`
- `@xterm/xterm`: `6.0.0`

结论：

- Electron 明显落后，值得评估升级。
- `node-pty` 和 `xterm` 不是“版本明显落后”的问题。
- `xterm` 还是自定义 tarball，升级成本高于普通依赖。

## 3. 走查结论

### 3.1 根因优先级判断

当前卡顿更像是代码结构问题叠加，而不是单纯 Electron 版本老：

1. 终端窗口一旦打开过就长期保留挂载，至少 `Paused/Stopped` 窗口没有被及时卸载；运行中的隐藏窗口是否需要长期保活，需要单独量化判断。
2. 渲染层对全局 store 的订阅粒度过粗，运行态小更新会放大为大范围重渲染。
3. 运行态会话数据和持久化工作区数据耦合过深。
4. 启动链路里有明确的人为延迟，且生产构建关闭了压缩。
5. `backgroundThrottling: false` 全局开启，会放大隐藏内容的后台成本。

因此：

- 只升级 Electron 可能会有改善。
- 但如果不先处理上述结构性问题，输入卡顿和长时间运行后的迟滞不会根治。

### 3.2 关键证据点

- `src/renderer/App.tsx`
  - 历史打开过的终端窗口会持续保留在 `mountedTerminalWindowIds` 中。
- `src/renderer/components/TerminalView.tsx`
  - 隐藏窗口仍保留完整终端视图、侧边栏、对话框和布局树。
- `src/renderer/components/TerminalPane.tsx`
  - 每个 pane 都维护 xterm 实例、scrollback、ResizeObserver、窗口 resize 监听、PTY 数据订阅。
- `src/renderer/stores/windowStore.ts`
  - `updatePane` 直接更新 layout tree；运行态字段和持久化字段共用一个 store。
- `src/main/index.ts`
  - `renderer-ready` 后仍有额外显示延迟，且 `backgroundThrottling` 被关闭。
- `vite.config.ts`
  - 生产构建 `minify: false`。

## 4. 修改事项评估

下面按“建议事项 -> 价值 -> 风险 -> 负面影响 -> 建议”整理。

### 4.1 优先卸载 `Paused/Stopped` 窗口，再评估运行中隐藏窗口的保活策略

建议：

- 不再让 `Paused/Stopped` 窗口长期保留在 React 树中。
- 运行中的隐藏窗口先保持现状，不做一刀切卸载。
- 对运行中的隐藏窗口先补采样指标，再决定是否需要“有限保活”。
- 如果后续要对运行中的隐藏窗口做卸载，重新进入窗口时使用现有 PTY history replay 恢复可见内容。

价值：

- 这是最稳妥的第一步，能先降低不必要的挂载成本。
- 对已经暂停的窗口，卸载不会影响会话连续性。
- 能先验证“前端挂载数量”是否确实是主要压力源。

风险等级：

- 中

主要风险：

- `Paused/Stopped` 窗口重新打开时，需要重新挂载前端视图。
- 如果后续扩展到运行中的隐藏窗口，才会引入 replay 时序、焦点、IME 等更高风险问题。

潜在负面影响：

- 已暂停窗口重新进入时，首次恢复可见内容的速度可能略慢于当前“始终挂着”的模式。
- 这一步对运行中的窗口切换速度不应产生明显负面影响。

建议：

- 第一优先级执行。
- 先限定在 `Paused/Stopped` 窗口。
- 对运行中的隐藏窗口，先记录数量、输出速率、切换耗时，再决定是否继续收紧保活策略。

### 4.2 把渲染层 store 订阅改成细粒度 selector

建议：

- 清理 `useWindowStore()` 直接整库订阅。
- `App`、`TerminalView`、`GroupView`、`Sidebar`、`useWindowSwitcher` 优先改造。
- action 尽量通过 `useWindowStore.getState()` 或单独 selector 获取。

价值：

- 这是改善“输入时掉帧”和“状态一变整页抖动”的第二关键项。
- 也会降低分屏、SSH 状态变化、tmux 元数据变化时的不必要渲染。

风险等级：

- 中高

主要风险：

- selector 如果写错，容易出现 UI 不刷新或刷晚一拍。
- 某些现有逻辑默认依赖“整库订阅后自然刷新”，拆细后可能暴露隐藏耦合。
- 过度 memo 化会让代码可读性下降。

潜在负面影响：

- 代码会更分散，调试路径比现在更长。
- 如果 equality 判断不当，会引入“看似随机”的渲染问题。

建议：

- 必须谨慎推进，先分析透热路径、订阅关系和渲染边界，再动手。
- 与 4.1 并行推进，但不能一次性全仓 sweep。
- 先改最热路径，建议顺序为 `App` -> `TerminalView` -> `Sidebar` -> `GroupView` -> `useWindowSwitcher`。
- 每改完一层就单独回归，确认窗口切换、分屏、组视图、SSH、tmux 状态更新都正常，再进入下一层。
- 对这条改动要优先补测试，尤其是“状态变化后 UI 仍能正确刷新”的测试。

### 4.3 拆分运行态 session store 与持久化 workspace store

建议：

- 把 `pid`、`status`、`sessionId`、tmux 元数据、Git 分支、Claude runtime 状态、SSH runtime cwd 等会话态从持久化树里抽出来。
- 工作区树只保留可恢复、可持久化的数据。

价值：

- 能从根上减少运行期状态写放大。
- 能降低 auto-save、layout 更新、窗口列表重算的频率。

风险等级：

- 高

主要风险：

- 这是结构性调整，影响恢复、保存、视图同步、事件分发。
- 容易在暂停/恢复/关闭窗口时引入状态不一致。

潜在负面影响：

- 开发复杂度明显上升。
- 短期内会让代码过渡层变多，不会立刻变简单。

建议：

- 放在 4.1、4.2 收益稳定之后做，不建议紧跟第二条立即启动。
- 先做局部试点，再决定是否推进完整拆分。
- 采用“双写过渡 + 最终切换”方式，不要一次性硬切。

### 4.4 把 SSH 运行时 cwd 变更改成 runtime-only

建议：

- SSH pane 的运行时 cwd 不要在每次变化时都通过持久化路径写回 workspace store。
- 运行时目录变化应优先更新 runtime 状态，保证 UI 立即正确。
- 如果确实需要恢复远端目录，应单独定义“可持久化 remote cwd 更新策略”，只在明确时机落盘。
- 建议的落盘时机是：暂停窗口、归档窗口、显式关闭窗口，或提供设置项控制“是否记住 SSH 最后目录”。

价值：

- 能减少输出解析和输入时触发的附带更新。
- 可降低 auto-save 频率和 layout 变更次数。

风险等级：

- 中

主要风险：

- 用户可能期望“下次恢复回到刚刚切到的目录”，如果直接改成纯 runtime-only，这个行为会变化。
- 如果运行态与持久化态边界定义不清，容易出现 UI 显示一个目录、恢复后回到另一个目录的认知落差。

潜在负面影响：

- 如果持久化时机设计不当，SSH 窗口恢复时的工作目录连续性可能下降。

建议：

- 作为 4.3 的前置小改动可以先做。
- 不建议简单做成“完全不保存 cwd”。
- 推荐方案是“运行时只走 runtime，持久化只在明确时机落盘”。
- 文档和设置项里要明确“SSH 目录是否持久化”的规则。

### 4.5 优化启动路径，移除人为延迟

建议：

- 缩短或移除 `renderer-ready` 前后的固定延迟。
- 重新评估淡入动画是否值得保留。
- 把 `SettingsPanel`、`QuickSwitcher`、`SSHSftpDialog` 等非首屏必需内容做懒加载。

价值：

- 能直接改善“打开应用慢”“首次打开终端慢”的主观感受。

风险等级：

- 中

主要风险：

- 之前为了掩盖白闪和布局跳动才加延迟，移除后旧问题可能回归。
- lazy load 后，首次打开某些弹层会有一次额外加载延迟。

潜在负面影响：

- 如果过度追求快显示，可能换来更明显的闪烁或样式未稳定。

建议：

- 放到最后单独评估，不作为近期优先项。
- 这类逻辑大概率是为了解决历史启动渲染问题引入的，不能直接删除。
- 等前面的结构性热点收敛后，再重新验证这些延迟和动画是否仍有必要。

### 4.6 重新验证生产构建开启压缩

建议：

- 重新验证 `minify: false` 的历史原因是否仍成立。
- 如果 xterm 运行时问题已经不存在，则恢复压缩构建。

价值：

- 可减少生产包 JS 体积和解析执行成本。
- 对冷启动和首次进入视图有直接帮助。

风险等级：

- 中

主要风险：

- 过去已经出现过“压缩后 xterm 运行时错误”的历史痕迹，说明这里不是零风险。

潜在负面影响：

- 如果复现旧问题，会导致生产环境终端直接不可用。

建议：

- 放到最后单独验证，不作为近期优先项。
- 必须先做单独验证，不要和其他大改混在一起。
- 可以先尝试 `esbuild` 压缩，再试 `terser`，逐步确认问题边界。

### 4.7 恢复或细化 `backgroundThrottling`

建议：

- 不要全局 `backgroundThrottling: false`。
- 优先恢复默认行为，必要时仅对特定场景保留例外。

价值：

- 对隐藏窗口、隐藏 webview、后台不活动 pane 的 CPU 与渲染负担有帮助。

风险等级：

- 中

主要风险：

- 某些后台逻辑如果错误依赖高频帧驱动，切回默认后会显得“恢复慢半拍”。

潜在负面影响：

- 背景 tab、背景 SSH 面板、隐藏 browser pane 的更新及时性可能下降。

建议：

- 放到最后单独验证，不作为近期优先项。
- 这项配置很可能也承担过历史兼容职责，不能直接恢复默认。
- 需要专门回归切后台、再切回前台的交互体验。

### 4.8 下调状态轮询和事件模型的开销

建议：

- 减少 `StatusPoller` 的无差别轮询频率。
- 优先依赖 PTY 输出、进程退出事件驱动；轮询只做兜底。

价值：

- 会改善大窗口数、大 pane 数场景下的主进程持续负担。

风险等级：

- 中低

主要风险：

- 状态更新可能不如当前“频繁轮询”及时。

潜在负面影响：

- 某些边界状态可能更晚切到 `waiting`。

建议：

- 当前先不做。
- 这不是第一优先级，也不像当前输入卡顿的主因。
- 除非后续量化指标明确显示主进程状态轮询是主要负担，否则不进入近期主线。

### 4.9 升级 Electron 到 41.x

建议：

- 单独分支升级到 `electron@41.x`。
- 同时配套升级 `electron-builder`。

价值：

- Chromium、V8、Node 底座更新明显。
- Electron 官方近几个大版本修过窗口 resize、平台兼容和基础行为问题。
- 长期维护价值高。

风险等级：

- 高

主要风险：

- `node-pty` 是原生模块，Electron ABI 升级后必须重点验证。
- 自定义标题栏、`webviewTag`、IME、tmux、SSH、文件选择、窗口事件都可能出现兼容性回归。
- 打包、签名、CI、不同平台的差异可能放大。

潜在负面影响：

- 需要较长回归周期。
- 短期内会增加“不确定是不是业务 bug 还是框架升级回归”的排查成本。

建议：

- 值得做，但不能替代 4.1 和 4.2。
- 应在热点结构优化之后独立推进。
- 记录为后续要做事项。
- 必须单独分支推进，不与 React/Vite 升级或大规模结构重构绑在一起。

### 4.10 升级 `electron-builder`

建议：

- 与 Electron 升级配套进行，从 `24.x` 升到 `26.x`。

价值：

- 保持构建链和 Electron 主版本匹配，减少打包兼容性风险。

风险等级：

- 中

主要风险：

- 安装器、目标平台配置、产物结构可能有变化。

潜在负面影响：

- 发布流程可能需要同步调整。

建议：

- 跟 Electron 升级放同一批次，不要单独提前做。
- 记录为后续要做事项。

### 4.11 升级 React 到 19.x

建议：

- 作为后续现代化升级项，不作为本轮性能问题的优先修复手段。

价值：

- 未来生态和编译器方向更一致。

风险等级：

- 中高

主要风险：

- 某些第三方库、旧写法、副作用时序在 React 19 下表现可能不同。

潜在负面影响：

- 如果和性能整改一起做，回归定位会非常混乱。

建议：

- 记录为后续要做事项。
- 等前面性能热点收敛后再做，且单独安排回归周期。

### 4.12 升级 Vite 到 8.x

建议：

- 与 React 升级分离看待，偏构建链现代化。

价值：

- 构建工具链更新、插件生态更新。

风险等级：

- 中

主要风险：

- 插件和构建配置兼容性变更。

潜在负面影响：

- 开发环境和打包配置需要重新验证。

建议：

- 记录为后续要做事项。
- 与 React 升级同批次规划，但不建议和 Electron 升级绑成一次性大升级。

### 4.13 升级 xterm custom 包

建议：

- 当前不纳入“后续版本升级都做”的同一批次。
- 原因不是不想做，而是当前上游 registry 仍是 `6.0.0`，没有明确版本差。
- 如果后续 xterm 上游出现新版本，或决定重新整理 custom patch，再单独评估并推进。

价值：

- 当前上游 registry 仍是 `6.0.0`，直接版本红利有限。

风险等级：

- 高

主要风险：

- 仓库使用的是 custom tarball，不是标准 npm 包。
- 任何升级都意味着要重新迁移自定义 patch。

潜在负面影响：

- 输入法、链接识别、重放行为、补丁兼容性都可能回归。

建议：

- 除非后续确认 custom patch 本身导致性能问题，否则先不动。

## 5. 推荐推进顺序

### Phase 0：先加量化指标

- 增加窗口切换耗时、终端恢复耗时、首次打开耗时、隐藏窗口数量、挂载 xterm 数量日志。
- 加一轮 React Profiler 和 Chromium Performance trace。

风险：

- 低

### Phase 1：先解决最可能的根因

- 4.1 优先卸载 `Paused/Stopped` 窗口
- 4.2 细化 store 订阅
- 4.4 SSH cwd runtime-only

风险：

- 中到高

收益：

- 最可能直接改善“输入卡顿”和“长时间运行后变重”。

### Phase 2：补运行中隐藏窗口观测

- 为运行中的隐藏窗口增加观测指标
- 决定是否需要“当前窗口 + 最近 N 个运行窗口”有限保活

风险：

- 中

收益：

- 为后续是否继续收紧运行中隐藏窗口保活提供依据。

### Phase 3：结构整理

- 4.3 拆 runtime store
- 4.8 重新设计状态更新路径

风险：

- 高

收益：

- 长期可维护性和性能边界都会更稳。

### Phase 4：依赖升级

- 4.9 Electron 41.x
- 4.10 electron-builder 26.x
- 4.11 React 19.x
- 4.12 Vite 8.x
- 4.13 xterm custom 包暂不纳入同一批次，等待版本差或 patch 整理时再单独评估

风险：

- 高

收益：

- 底层平台能力和长期维护收益。

### Phase 5：最后验证历史兼容逻辑

- 4.5 启动延迟优化
- 4.6 重新验证压缩构建
- 4.7 恢复或细化 background throttling

风险：

- 高

收益：

- 有机会继续优化冷启动、后台负担和整体体感。
- 但这几项都带有较强历史兼容背景，应在最后单独验证。

## 6. 不建议的推进方式

- 不建议一口气同时做“结构重构 + Electron 升级 + React 升级”。
- 不建议先升级 React/Vite 再回头处理挂载和订阅问题。
- 不建议在没有量化指标前就只靠主观体感判断是否优化成功。

## 7. 总结

结论很明确：

- 当前最值得先做的是：
  1. 先卸载 `Paused/Stopped` 窗口。
  2. 细化 store 订阅。
  3. 对运行中的隐藏窗口补数据，再决定是否进一步收紧保活策略。
- 运行中的窗口会话不应为了性能被直接停止。
- Electron 升级值得做，但它不是本轮卡顿的唯一答案。
- 运行态与持久化态耦合，是后续必须处理的结构问题。
