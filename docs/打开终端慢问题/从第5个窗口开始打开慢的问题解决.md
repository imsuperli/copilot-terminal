# 从第5个窗口开始打开慢的问题解决

## 现象

在主界面依次点击窗口卡片进入终端时：

- 前几个窗口通常打开很快
- 从第 5 个左右开始，进入终端后要额外等待约 3 到 4 秒
- 慢的时候界面会卡在 PowerShell 启动前，随后才出现：

```text
PowerShell 7.5.4
PS D:\sources\idea_projects\tjyhy\tjcm-user>
```

## 结论

这个问题表面上看像是“窗口开多了以后资源争抢”，但根因并不是多个终端不独立，也不是第 5 个这个数字本身有特殊含义。

真正的根因是：

- 终端进程早就已经在后台启动
- 用户第一次点击卡片进入某个窗口时，渲染层才第一次挂载对应的 `xterm.js`
- 挂载时会先做一次 PTY history replay
- 这次 replay 期间，代码为了避免旧历史触发 synthetic reply，把 `onData -> ptyWrite` 临时整体屏蔽了
- 但 PowerShell 启动历史里本来就包含终端能力查询 `ESC[c`
- `xterm.js` 重放到这段数据时会生成 DA 响应
- 响应被屏蔽后，PowerShell 只能等待约 3 秒超时，再继续启动

所以慢的不是“第 5 个窗口”，而是“第一次真正挂载 xterm 的窗口”。之所以现象上经常表现成“前几个快、后几个慢”，是因为应用会保留已经打开过的 `TerminalView` 挂载状态，之前打开过的窗口再次进入会很快，而后面还没真正 attach 的窗口会集中暴露这个问题。

## 为什么每个终端明明是独立的，还是会出现这个现象

每个窗格/窗口对应的 PTY 进程本身确实是独立的，这一点没有问题。

问题发生在 shell 与终端模拟器之间的协议握手，不在 PTY 进程隔离上：

- PowerShell 会向“终端模拟器”发能力查询
- `xterm.js` 负责生成协议响应
- 渲染层代码负责把这个响应回写给 PTY

只要这个回写链路被挡住，独立的 PTY 一样会卡住等待超时。

## 这次回归和之前修复的关系

之前已经修过一次非常相似的问题：当 `onData` 里带有 `isActiveRef.current` 判断时，非激活窗格的协议响应会被直接丢掉，导致 PowerShell 启动慢。

这次不是 `isActive` 判断回来了，而是另一条新路径重新把协议响应挡住了：

- 旧问题：非 active pane 不转发 `onData`
- 这次问题：history replay 期间临时 suppress 了 `ptyWrite`

两次问题的共同点都是一样的：

- PowerShell 发出了 `ESC[c`
- `xterm.js` 生成了 DA 响应
- 响应没有成功回到 PTY
- PowerShell 等待超时约 3 秒

## 修复方案

修复后的策略不是简单地完全取消 suppress，而是按“会话阶段”区分：

- 当前会话的首次 history replay：允许协议响应正常回写
- `pid` 切换后的新会话 replay：同样允许协议响应正常回写
- 同一会话后续的补回放：继续 suppress synthetic reply，避免把旧历史里的控制查询再次注入 live PTY

实现上增加了 `hasCompletedReplayForCurrentSessionRef`，专门区分：

- 这是当前会话第一次回放
- 还是当前会话已经回放过一次，后面只是补回放

关键修改位置：

- `src/renderer/components/TerminalPane.tsx`

## 为什么这个修复是安全的

这个修改不会让后台窗口误发用户输入，原因是：

- `xterm.js` 的普通用户键盘输入只有在终端拿到焦点时才会产生
- 未激活、未聚焦的窗格不会凭空产生真实用户输入
- 当前放开的只是“首次/新会话回放阶段的协议响应”
- 同一会话后续补回放仍然保留 suppress 保护

换句话说，这次修复恢复的是 shell 启动必须依赖的协议握手，不是放开任意后台输入。

## 验证结果

修复后，表现恢复为：

- 依次打开多个窗口时，每个窗口首次进入都能快速显示提示符
- 不再出现从第 5 个窗口开始稳定增加 3 到 4 秒等待的问题

本次还同步补了回归测试思路：

- 初次 history replay 中如果包含 `ESC[c`
- `xterm.js` 生成的 DA 响应必须成功调用 `ptyWrite`

## 后续维护注意事项

后面如果继续优化以下能力，需要特别小心：

- PTY history replay
- pane/window 首次挂载
- hidden terminal 保持挂载
- synthetic reply 过滤
- `onData` 转发条件

原则只有一条：

> 只要是 shell 正在等待的终端协议响应，就必须保证能回到 PTY，不能因为 active 状态、隐藏状态、回放状态而被误伤。
